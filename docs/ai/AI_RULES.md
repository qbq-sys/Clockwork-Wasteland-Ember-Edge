# AI 开发规则

## 当前最高目标

当前阶段优先完成 Playable Slice 0：第一份悬赏。

优先级：
1. 战斗可玩
2. 站位规则成立
3. 技能合法性清楚
4. Bulwark + Executioner 战斗循环成立
5. 悬赏 → 战斗 → 结算 → 返回据点闭环成立

暂时不要主动扩展：
- 完整剧情对话树
- 拍卖式招募系统
- 完整基地经营
- 装备系统
- 更多职业
- 复杂职业树
- 大规模重构

---

## 代码修改原则

1. 每次只做一个明确小任务。
2. 不要大规模重构，除非明确要求。
3. 不要为了“完整性”提前建设复杂框架。
4. 新功能优先接入正式链路，不继续扩大旧原型链路。
5. 如果发现旧代码和当前设计冲突，先报告，不要擅自推翻。
6. 修改前先说明计划，修改后说明改了哪些文件。
7. 不要主动新增未要求的系统、字段、UI 页面或玩法。
8. 无法确认的地方先做最小实现，不要扩大需求。

---

## UI 规则

1. UI 结构优先在可编辑资源中完成。
2. 运行时代码只负责绑定、刷新、显示、隐藏和交互。
3. 页面脚本只处理本页面逻辑。
4. UI 管理器只负责页面切换和生命周期。
5. 影响玩家判断的信息必须有明确显示出口。

---

## 数据规则

1. 配置数据和运行逻辑分离。
2. 技能、角色、敌人、文本、数值优先数据驱动。
3. 不在代码里长期硬编码正式内容。
4. 同一概念在代码、资源、文档中命名保持一致。

---

## 系统规则

1. 核心流程由专门控制层调度。
2. 页面、表现对象、数据对象不要接管主流程。
3. 状态复杂后要显式化，不要散落在多个函数里。
4. 通用系统只在真实需求出现后再抽象。
5. 不允许万能 Manager 无限膨胀。

---

## 每次任务完成后必须输出

1. 修改文件列表
2. 新增或修改的类  函数
3. 是否影响现有系统
4. 如何测试
5. 测试结果，或无法测试的原因
6. 下一步建议
