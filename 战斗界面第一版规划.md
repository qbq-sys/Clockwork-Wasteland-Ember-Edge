# 战斗界面第一版规划

用途：用于约束第一版战斗 HUD / 战斗信息界面的信息结构、显示优先级与交互表达，指导后续 UI 设计稿、Prefab 结构和战斗逻辑接线。

## 一、目标

战斗界面的核心目标不是堆叠信息，而是帮助玩家快速完成以下判断：

1. 现在轮到谁行动
2. 当前行动角色能用什么技能
3. 该技能从哪个站位可以释放
4. 该技能可以命中敌方哪些站位
5. 当前场上谁危险、谁被控、谁有增益或减益
6. 当前选中的目标是否值得优先处理

因此，战斗 HUD 的信息组织必须围绕以下六个核心维度展开：

- 站位
- 技能可用性
- 目标合法性
- 生命状态
- Buff / Debuff
- 被动与专精特征

## 二、战斗界面必须显示的信息

### 2.1 顶部战斗状态区

必须显示：

- 当前回合数
- 当前行动方提示
- 当前行动角色名称
- 当前行动角色头像或小立绘

目的：

- 让玩家第一眼明确当前操作权归属

### 2.2 战场中的单位头顶信息

每个场上单位必须显示：

- 名字
- 当前生命 / 最大生命
- 当前站位编号 `1 / 2 / 3 / 4`
- Buff / Debuff 图标
- 濒危 / 倒地 / 尸体状态提示

目的：

- 不打开详情也能直接判断场面危险度和站位关系

### 2.3 当前行动者信息面板

必须显示：

- 名字
- 职能 / 职业
- 专精
- 偏好站位
- 当前生命 / 最大生命
- 当前资源 / 最大资源
- 攻击
- 防御
- 速度
- 当前站位
- 被动技能名称
- 若存在成长被动，则一并显示成长被动名称

说明：

- 被动技能要显示，但不应长期占用最大面积
- 常驻显示被动名称
- 详细说明进入 tooltip 或次级说明区域

### 2.4 技能列表区

每个技能按钮必须显示：

- 技能图标
- 技能名称
- 资源消耗
- 冷却回合
- 技能简短描述
- 技能是否可用

若技能不可用，必须能体现原因：

- 资源不足
- 冷却中
- 当前站位不可释放
- 当前没有合法目标

说明：

- 技能列表是战斗中最核心的信息区之一

### 2.5 技能站位说明区：8 点规则

每个技能都必须带一组固定的 `8 点站位说明`。

规则固定如下：

- 左边 4 个点：表示施法者可站位
- 右边 4 个点：表示目标可命中站位
- 每边从左到右固定表示 `1 2 3 4`

示例：

- 如果技能只能从 `2、3` 号位释放，则点亮左侧第 `2、3` 个点
- 如果技能只能命中敌方 `2、3` 号位，则点亮右侧第 `2、3` 个点

表现规则：

- 可用位置点亮
- 不可用位置熄灭或暗灰
- 群体技能可同时点亮多个点
- 全体技能可点亮右侧全部 4 点
- 对自身或友方生效的技能，也必须遵循统一站位表达逻辑

要求：

- 该 8 点表达属于技能核心信息，不是装饰元素
- 后续 Skill 按钮 prefab 必须预留这组结构

### 2.6 当前选中目标详情区

当玩家点击或选中某个单位时，必须显示：

- 名字
- 我方 / 敌方
- 职能 / 专精
- 当前生命 / 最大生命
- 当前资源
- 攻击 / 防御 / 速度
- 当前站位
- 偏好站位
- 被动技能名称
- Buff / Debuff 列表
- 当前状态描述

状态描述至少应覆盖：

- 眩晕
- 燃烧
- 中毒
- 护卫
- 濒危
- 倒地

建议同时显示：

- 专精特征一句话总结
- 被动技能一句话说明

### 2.7 Buff / Debuff 信息

战斗中必须清楚显示状态效果，至少包含：

- 图标
- 名称
- 剩余回合
- 是否控制类
- 是否持续伤害类
- 是否增益 / 减益

建议：

- 角色头顶显示简化图标
- 目标详情区显示完整说明
- 悬停状态图标时显示 tooltip

### 2.8 战斗日志区

必须显示最近发生的关键行动：

- 谁对谁使用了什么技能
- 造成多少伤害
- 回复多少生命
- 附加了什么状态
- 哪个状态触发了什么效果
- 谁被击倒 / 死亡 / 恢复

说明：

- 日志是帮助玩家理解战斗过程的解释层，不是装饰

## 三、被动技能信息显示规则

### 3.1 常驻显示规则

在以下区域必须显示被动技能名称：

- 当前行动者信息面板
- 当前选中目标详情区

若同时存在基础被动与成长被动，则按如下规则显示：

- 基础被动：`xxx`
- 成长被动：`xxx`

### 3.2 详细说明规则

被动的完整说明不要求长期常驻展开，但必须有明确的信息出口，可采用以下任一方式：

- tooltip
- 次级说明文本区域
- 展开说明面板

### 3.3 触发反馈规则

当被动触发时，建议至少给出一种明确反馈：

- 战斗日志提示
- 飘字提示
- 小图标闪动
- 短文本提示

目标：

- 让玩家不仅“看得到被动名称”，还要“感知到被动正在发挥作用”

## 四、信息层级

### 4.1 一级常驻信息

一级常驻信息必须优先保证可读性：

- 当前行动者
- 回合数
- 生命
- 资源
- 技能列表
- 技能 8 点站位说明
- 当前目标信息
- Buff / Debuff 图标
- 倒地 / 濒危状态

### 4.2 二级常驻信息

二级信息可以弱化，但需要保持可见：

- 职能
- 专精
- 偏好站位
- 被动技能名称
- 技能冷却
- 战斗日志

### 4.3 三级信息

三级信息建议通过悬停、选中或展开显示：

- 被动详细说明
- 专精详细说明
- Buff / Debuff 完整描述
- 技能详细描述
- 技能特殊机制说明

## 五、推荐布局结构

第一版战斗界面建议按以下逻辑组织：

1. 顶部
   - 回合数
   - 当前行动者提示

2. 左侧或左下
   - 当前行动者详情
   - 包含被动名称、专精、生命、资源、属性、站位

3. 中下区域
   - 技能栏
   - 每个技能按钮必须带 8 点站位说明

4. 右侧
   - 当前选中目标详情
   - 包含被动、状态、专精、站位、属性

5. 战场单位头顶
   - 名称
   - 血量
   - 站位
   - 状态图标

6. 底部或侧边
   - 战斗日志

## 六、视觉与风格要求

战斗界面整体视觉应符合项目气质：

- 工业废土
- 机械感
- 战术感
- 清晰
- 压迫感
- 强信息可读性

明确避免：

- 纯幻想发光 UI
- 过度科幻全息蓝屏风格
- 可爱、圆润、轻飘的视觉语言

推荐表现方向：

- 金属铭牌
- 机械槽位
- 战术仪表
- 磨损面板
- 工业铆钉与指示灯

技能 `8 点站位说明` 的视觉表现可以借鉴成熟前后排回合制的表达逻辑，但视觉语言必须转化为本项目风格，例如：

- 金属铆钉点
- 机械指示灯
- 刻度槽位
- 战术灯列

## 七、实现约束

后续落地时应遵循以下约束：

1. 战斗 HUD 设计必须优先服务信息判断，不以装饰优先
2. 技能站位说明必须标准化，不得每个技能使用不同表达方法
3. 被动信息必须提供常驻名称与详细说明两层表达
4. 任何影响玩家判断的战斗状态，必须有 UI 出口
5. Prefab 中应为技能按钮、目标详情和状态说明预留稳定节点，不得依赖运行时临时拼接大块结构

## 八、一句话设计摘要

请把第一版战斗界面设计成一个强调 `站位、技能合法性、状态联动、被动特征` 的回合制战斗 HUD。

其中必须明确表现：

- 当前行动者
- 技能可用性
- 技能 8 点站位说明
- 当前目标详情
- Buff / Debuff
- 被动技能信息
- 战斗日志
