# 项目级补充规范

本文件用于补充 [开发规范.md](D:/Project/Clockwork Wasteland Ember Edge/开发规范.md) 中未覆盖的项目专属约束。

适用范围：
- 当前项目全部运行时代码
- 当前项目全部 UI Prefab
- 当前项目相关编辑器工具与自动化补丁

---

## 一、UI 架构补充规则

### 1.1 当前项目 UI 正式链路

当前项目 UI 正式链路统一为：

`BattleController -> BattleUI(兼容门面) -> UIManager -> 各 Screen / HUD / Popup`

要求：
- `BattleController` 负责业务流程调度
- `BattleUI` 只保留兼容入口和转发职责
- `UIManager` 负责界面生命周期和切换
- 各 `XXXUI.cs` 负责各自页面的绑定、显示和交互事件

### 1.2 BattleUI 只允许作为兼容门面存在

规则：
- `BattleUI` 可以保留，但只允许承担兼容转发职责
- 后续新功能不得继续接入 `BattleUI` 的旧运行时造界面链路

禁止：
- 在 `BattleUI` 中继续新增整页运行时构建逻辑
- 在 `BattleUI` 中继续新增正式页面实现

要求：
- 新页面逻辑进入 `UIManager + 独立 XXXUI.cs`
- 已迁出的页面不再回接旧 builder 代码

### 1.3 UIManager 只负责界面切换和生命周期

规则：
- `UIManager` 只负责：
  - screen / hud / popup 的创建
  - 缓存
  - show / hide
  - root 挂载
  - 统一切换入口

禁止：
- 在 `UIManager` 中堆页面内部业务逻辑
- 在 `UIManager` 中扩展页面布局生成逻辑

### 1.4 每个界面必须有独立控制文件

规则：
- 每个页面必须有独立 `XXXUI.cs`
- 页面正式实现不得长期堆在公共总文件中

当前页面范围包括但不限于：
- `StartMenuUI`
- `LobbyUI`
- `SettingsUI`
- `SaveSlotUI`
- `TavernUI`
- `AdventureMapUI`
- `TeamSelectionUI`
- `HeroCodexUI`
- `PopupUI`
- `BattleHudUI`
- `SkillDescriptionUI`
- `LevelUpUI`

要求：
- 页面脚本只处理本页逻辑
- 页面迁出后，旧实现应尽快冻结并清理

## 二、Prefab 驱动补充规则

### 2.1 当前项目所有正式页面必须 Prefab 驱动

规则：
- 页面结构、位置、尺寸、图片、文本样式、按钮布局，全部以 Prefab 为准
- 运行时代码只负责绑定、刷新、显示和交互

禁止：
- 在运行时代码中重建完整页面结构
- 在运行时代码中重写 Prefab 已确定的布局参数

### 2.2 运行时代码不得控制固定布局位置

规则：
- 非必要情况下，按钮位置、图片位置、容器位置、文本位置都按 Prefab 摆放结果使用
- 代码不负责写死固定坐标

允许：
- 运行时控制显示和隐藏
- 运行时刷新文本和图像
- 基于模板生成动态列表项

禁止：
- 用代码重新安排整页组件坐标
- 用代码覆盖人工调好的布局

### 2.3 动态内容必须来自模板

规则：
- 列表项、技能项、地图项、英雄项、图标项等动态内容必须基于模板节点或独立 item prefab 克隆

禁止：
- 用 `new GameObject(...)` 临时拼出正式样式项
- 在代码里重建动态项主布局结构

### 2.4 页面绑定优先显式引用

规则：
- 固定按钮、文本、容器、模板节点优先使用 `[SerializeField]`
- 名字查找、文案查找只允许作为临时过渡兜底

要求：
- Prefab 稳定后逐步移除文案匹配绑定
- 关键交互节点最终必须变成显式字段

## 三、导航与页面交互补充规则

### 3.1 页面返回默认返回来源页

规则：
- 从多个入口可进入的页面，返回目标必须由调用方显式提供
- 页面内部不得默认假定固定返回目标

推荐形式：
- `ShowSettings(Action onBack, ...)`
- `ShowSaveSlots(..., Action onBack)`

### 3.2 页面只派发事件，不直接接管总流程

规则：
- 页面点击事件只负责回调调用方
- 页面内部不直接承担总流程编排职责

要求：
- 页面通过回调通知上层
- 上层控制器决定后续流程切换

### 3.3 多入口页面必须统一行为

规则：
- 同一页面从不同入口打开时，其基础显示逻辑应一致
- 入口差异只体现在数据和返回行为，不体现在页面硬编码分叉

---

## 四、编辑器工具补充规则

### 4.1 自动化工具默认增量补丁

规则：
- 本项目工具链优先修补已有 Prefab，而不是整份重建
- 已人工调整的布局和细节默认视为有效成果

禁止：
- 为了更新功能整体覆盖现有 Prefab
- 工具执行后抹掉人工微调结果

### 4.2 页面生成链与运行链分离

规则：
- 编辑器工具负责生成、修补、补节点
- 运行时代码负责绑定、显示、刷新

禁止：
- 让运行时代码承担 Prefab 修复职责
- 让页面脚本同时扮演运行时控制器和生成器

---

## 五、旧链路清理补充规则

### 5.1 已迁移页面不再扩展旧实现

规则：
- 一旦某个页面迁移到独立 `XXXUI.cs`，后续该页面功能不得继续回写旧实现

要求：
- 旧逻辑冻结
- 新逻辑只进新文件

### 5.2 双轨只能短期存在

规则：
- 旧链路和新链路允许短期并存
- 但必须明确哪条是正式链路

要求：
- 新链路承担后续功能
- 旧链路只保留过渡兼容
- 时机合适时删除旧链路

### 5.3 每次拆分以“可收口”为目标

规则：
- 每次页面拆分都必须满足：
  - 页面逻辑归位到独立文件
  - 页面布局不再由运行时代码重建
  - 页面入口仍可正常工作

禁止：
- 只拆文件，不收职责
- 只迁代码，不停止旧链路扩展

---

## 六、当前项目执行顺序

后续默认按以下顺序推进：

1. 先拆页面控制逻辑
2. 再去掉对应页面的运行时布局重建
3. 再把关键节点改成显式字段绑定
4. 再冻结并清理旧链路
5. 最后压缩 `BattleUI` 与公共总文件中的残留实现

---

## 七、一句话总纲

**当前项目的 UI 必须逐步收口到 Prefab 驱动、页面独立、管理层统一、旧链路退场的新结构中，任何后续开发都不得继续扩散运行时代码造界面的旧模式。**
