# 游戏开发规范

本规范用于约束项目开发过程中的 UI、数据、流程、资源、工具链与技术架构实现方式。

适用范围：
- 当前项目
- 后续同类型项目
- 小型到中型团队协作开发

原则：
- 先定义规则，再扩展功能
- 先收口正式链路，再继续堆功能
- 先保证可维护，再追求局部方便

---

## 一、通用工程规则

### 1.1 正式链路优先

规则：
- 所有新增功能必须优先接入正式开发链路
- 原型方案只能作为短期过渡
- 一旦正式方案确定，后续功能不得继续接入旧原型链路

要求：
- 每个模块都应明确“正式实现入口”
- 同一功能不得长期存在双轨实现

### 1.2 单一职责

规则：
- 每个类、组件、脚本、工具文件应只负责一类核心职责
- 管理层、表现层、数据层、工具层不得长期混写

要求：
- 控制器只负责流程和调度
- 页面脚本只负责页面本身
- 数据对象只负责数据表达
- 工具脚本只负责工具行为

### 1.3 先约束边界，再开始编码

规则：
- 开发新模块前，应先确定其职责边界、输入输出和依赖关系
- 不允许在边界未定义清楚时直接堆功能

最低要求：
- 这个模块负责什么
- 不负责什么
- 谁调用它
- 它调用谁

### 1.4 命名必须服务维护

规则：
- 命名必须稳定、清晰、可搜索
- 正式代码不得保留临时性、试验性、含糊性命名

要求：
- 类名表达职责
- 方法名表达动作
- 变量名表达含义
- 路径命名保持一致风格

---

## 二、UI 开发规范

### 2.1 UI 结构以可编辑资源为准

规则：
- UI 的结构、层级、位置、尺寸、样式应以可编辑资源为准
- 运行时代码只负责绑定、显示、刷新和交互

要求：
- 界面布局应在编辑器资源中完成
- 运行时代码不负责重建整页布局

### 2.2 每个界面独立控制

规则：
- 每个完整页面应有独立控制脚本
- 页面逻辑不得长期集中在单一总文件中

要求：
- 一个页面对应一个主要页面类
- 页面类只处理本页逻辑

### 2.3 页面脚本只负责页面行为

规则：
- 页面脚本只负责：
  - 绑定节点
  - 刷新显示
  - 派发交互事件
  - 维护本页状态

禁止：
- 在页面脚本中承担全局导航职责
- 在页面脚本中承担资源生成职责
- 在页面脚本中承担上层业务编排职责

### 2.4 UI 管理层只负责切换和生命周期

规则：
- UI 管理器只负责页面创建、缓存、显示、隐藏、切换和挂载
- UI 管理器不负责页面内部逻辑

要求：
- 页面切换入口统一
- 生命周期管理统一
- 页面业务逻辑留在页面脚本内部

### 2.5 页面返回规则必须统一

规则：
- 页面返回默认表示返回调用来源
- 多入口页面的返回目标必须由调用方显式提供

要求：
- 不在页面内部写死固定返回目标
- 页面返回逻辑必须可预测

### 2.6 动态内容基于模板实例化

规则：
- 列表项、按钮项、卡片项、图标项等动态内容应基于模板或独立资源实例化
- 不在运行时临时拼装正式样式结构

要求：
- 动态内容可以生成
- 界面结构和样式应预先定义

### 2.7 页面绑定优先显式引用

规则：
- 固定节点优先使用显式字段绑定
- 名字扫描、文案匹配、层级遍历只允许短期兜底

要求：
- 关键按钮、文本、容器、模板节点使用显式引用
- 页面资源稳定后移除不稳定兜底绑定

## 三、数据与配置规范

### 3.1 数据与逻辑分离

规则：
- 配置数据与运行逻辑必须分离
- 数据对象不承担复杂流程逻辑

要求：
- 静态配置独立存放
- 运行时状态单独管理
- 行为逻辑由系统或控制器处理

### 3.2 配置优先

规则：
- 可配置内容优先做成数据驱动
- 不在代码中长期硬编码正式内容

适用对象：
- 数值
- 文本
- 描述
- 技能参数
- 角色参数
- UI 显示内容

### 3.3 数据结构服务正式成长与扩展

规则：
- 数据结构设计必须考虑后续成长、扩展、存档和读取需求
- 临时数据模型不得长期阻碍正式系统接入

要求：
- 结构可扩展
- 字段含义明确
- 版本演进可控

### 3.4 配置命名统一

规则：
- 同类配置采用统一命名方式
- 同一概念在代码、资源、文档中保持一致

要求：
- 不同模块不重复发明同义命名
- 正式术语统一

---

## 四、流程与系统设计规范

### 4.1 流程控制集中

规则：
- 核心流程应由专门控制层统一调度
- 不允许多个下层对象各自决定主流程跳转

要求：
- 页面不直接接管总流程
- 表现对象不直接编排主流程
- 主流程入口清晰

### 4.2 状态必须逐步显式化

规则：
- 当流程复杂度上升时，状态必须显式表达
- 不允许长期依赖隐式状态散落在多个函数中

要求：
- 大流程状态清晰
- 子流程状态清晰
- 状态切换入口清晰

### 4.3 跨系统依赖必须克制

规则：
- 系统之间优先保持明确依赖方向
- 不允许任意模块相互直接耦合

要求：
- 上层调度下层
- 下层通过明确接口返回结果
- 避免横向随意调用

### 4.4 基础系统按真实需求渐进引入

规则：
- 对象池、事件系统、状态机、服务层等基础系统应按实际需求逐步引入
- 不为“看起来完整”而提前建设大框架

要求：
- 先解决真实问题
- 先局部验证
- 再决定是否推广为通用能力

### 4.5 通用系统必须有明确职责

规则：
- 通用管理器、服务、工具系统必须明确职责边界
- 不允许发展为万能入口

要求：
- 一个服务只负责一类问题
- 通用系统必须有清晰命名和稳定职责

---

## 五、资源与 Prefab 规范

### 5.1 资源是正式资产

规则：
- 资源文件、Prefab、模板、配置表视为正式工程资产
- 不允许把资源当作一次性中间产物随意覆盖

要求：
- 资源结构稳定
- 资源命名清晰
- 资源修改可追踪

### 5.2 自动化工具默认增量修改

规则：
- 自动化工具优先做增量补丁
- 不得默认整体覆写已有人工调整内容

要求：
- 工具修改范围可控
- 工具输出可验证
- 工具运行结果可回滚

### 5.3 模板统一来源

规则：
- 同一类资源结构应优先来自统一模板来源
- 不同模块不得长期复制多份近似模板各自维护

要求：
- 模板职责明确
- 模板和实例关系清晰

---

## 六、玩家可读性规范

### 6.1 影响决策的信息必须可读

规则：
- 任何影响玩家选择、判断、构筑、战斗或资源分配的系统，都必须提供可读信息出口

要求：
- 效果可见
- 状态可见
- 条件可见
- 结果可理解

### 6.2 展示与实际效果保持一致

规则：
- UI 描述、配置文本、提示信息必须与真实逻辑一致
- 不允许长期存在“显示一套、结算一套”

要求：
- 配置、显示、逻辑统一
- 重要效果可验证

---

## 七、编辑器与工具链规范

### 7.1 工具服务生产，不替代正式结构

规则：
- 编辑器工具用于提高生产效率
- 工具不得成为运行时正式逻辑的替代物

要求：
- 编辑器做生成、修补、批处理
- 运行时做绑定、读取、执行

### 7.2 面向内容生产的工具应可读可用

规则：
- 面向策划、设计、调参使用的工具必须可读、易理解、低误用

要求：
- 术语统一
- 说明清晰
- 必要时提供中文化或注释说明

### 7.3 工具链应支持重复执行

规则：
- 正式工具应支持重复运行
- 不得依赖“一次性正确操作”才能工作

要求：
- 重复执行结果可预测
- 不破坏已有有效资源

---

## 八、技术债管理规范

### 8.1 原型代码必须可识别

规则：
- 原型实现、过渡实现、兼容实现必须可识别
- 正式实现和临时实现不得长期混淆

要求：
- 标明用途
- 标明保留原因
- 标明后续收口方向

### 8.2 旧链路有替代后必须收口

规则：
- 一旦正式替代稳定，旧链路必须降级、冻结或删除
- 不允许继续向旧链路追加正式功能

### 8.3 每次重构必须定义目标边界

规则：
- 每次重构前必须明确：
  - 哪些部分进入正式链路
  - 哪些部分暂时保留
  - 哪些部分后续删除

要求：
- 重构目标明确
- 过渡期清晰
- 收口点明确

---

## 九、协作与文档规范

### 9.1 规则优先于个人习惯

规则：
- 团队协作时，统一规则优先于个人编码习惯
- 实现方式应优先服从既定规范

### 9.2 文档只保留长期有效规则

规则：
- 开发规范文档应优先记录长期有效的规则、边界和约束
- 临时问题、局部例外、短期讨论内容不写入正式规范正文

### 9.3 重要模块必须有最小说明

规则：
- 核心系统、关键流程、通用工具应有最小可读说明

最低要求：
- 职责
- 入口
- 依赖
- 使用方式

---

## 十、执行原则

后续开发默认按以下顺序判断方案是否合格：

1. 是否接入正式链路
2. 是否职责单一
3. 是否边界清晰
4. 是否便于维护
5. 是否便于扩展
6. 是否便于内容生产
7. 是否便于玩家理解

如果不能满足上述原则，应先调整方案，再开始实现。

---

## 十一、一句话总纲

**资源决定结构，配置决定内容，流程决定调度，页面决定表现，系统按真实需求渐进收口，所有正式功能都必须进入可维护的正式链路。**
