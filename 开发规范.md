# 《发条荒野：余烬边境》开发规范

本规范用于约束后续 UI、编辑器工具、数据结构与运行时逻辑的实现方式。  
目标是减少返工、避免旧原型方式继续扩散，并保证项目逐步收敛到可维护、可生产的正式结构。

---

## 一、UI 开发规范

### 1. UI 外观必须以 Prefab 为准

规则：

- UI 的结构、样式、锚点、位置、尺寸、颜色、九宫格、字体，默认都在 Prefab 中维护
- 运行时代码只负责：
  - 找引用
  - 显示 / 隐藏
  - 填文本
  - 刷图标
  - 绑定点击事件
  - 控制状态切换

禁止：

- 运行时强制覆盖 Prefab 中已有的 `Image.sprite / color / type`
- 运行时重置已经在 Prefab 中手调过的 UI 外观

---

### 2. 禁止运行时整页代码生成 UI

规则：

- 不再允许通过 `CreatePanel / CreateText / CreateButton` 在运行时直接拼出整页界面
- Screen 级 UI（开始界面、大厅、图鉴、升级界面、设置、弹窗等）必须使用可编辑的 Prefab 结构

允许：

- 编辑器阶段生成或重建 UI Prefab
- 运行时在已有 Prefab 结构上绑定数据

禁止：

- `BuildLayout()` 在运行时 `ClearChildren()` 后重建整页 UI
- 改了 Prefab 后，运行时又被代码生成结构覆盖

---

### 3. 动态列表必须基于 Prefab 模板克隆

规则：

- 列表项、按钮项、技能项、目标项等动态内容，允许运行时生成实例
- 但这些实例必须来自 Prefab 中已有的模板节点

推荐方式：

- `SkillButtonTemplate`
- `HeroButtonTemplate`
- `BuffIconTemplate`
- `TargetButtonTemplate`

禁止：

- `new GameObject("RuntimeButton_xxx")` 再临时拼样式
- 运行时发明列表项结构和表现

结论：

**允许运行时生成“数据项实例”，不允许运行时生成“界面结构和样式”。**

---

### 4. UI 生成链与运行链必须分离

规则：

- 编辑器负责：
  - 创建 / 重建 Prefab
  - 生成模板结构
  - 补缺失节点
- 运行时负责：
  - 绑定
  - 显示
  - 刷新数据

推荐模式：

- `Build()`：运行时绑定，不重建结构
- `RebuildLayoutFromCode(...)`：仅编辑器 authoring / prefab 生成链使用

禁止：

- 一个类同时承担：
  - 运行时控制器
  - 运行时 UI builder
  - Prefab 修复器

---

## 二、Prefab 与自动化工具规范

### 5. 工具默认做增量补丁，不覆盖人工微调

规则：

- 对已有 Prefab 的自动化处理应优先采用增量修改
- 已经人工调好的内容不能轻易被工具覆盖

典型保护对象包括：

- `ActionOverlay` / `HitOverlay` 的 Transform
- 手工调过的 UI 锚点与布局
- 手工调过的 collider / sorting / offset

禁止：

- 因为“重建方便”而整体重写现有 Prefab
- 工具每次更新都抹掉人工校准

---

### 6. Feature Patch / Feature Template 统一走模板驱动

规则：

- 单位 Prefab 的新增功能节点（如 DebuffStatusBar、DialogueBubble、RageBar）必须通过 Feature Patch 系统接入
- 每个 Feature 必须有自己的模板 Prefab
- 同步逻辑永远从模板读取结构

目标：

- 旧 Prefab 自动补齐新功能
- 新 Prefab 自动带最新 Feature
- 不重建整个单位 Prefab
- 不覆盖不相关节点

---

## 三、编辑器工具规范

### 7. 面向设计和调参的界面默认中文化

规则：

- 技能编辑器、Buff 编辑器、战斗参数面板、角色管理器等面向设计/策划使用的界面，默认提供中文名称
- 必要时补充中文说明，避免只有英文内部字段

目标：

- 提高可读性
- 减少误用
- 降低工具使用门槛

---

### 8. 角色管理器必须服务正式成长系统

规则：

- 非敌方单位管理器不能只停留在“属性 + 技能 + 一个被动”
- 必须逐步扩展到支持：
  - 职业
  - 专精方向
  - 固定被动
  - 成长节点
  - 后续升级选择接口

要求：

- 数据结构优先服务成长与构筑系统
- 工具结构要跟上战斗系统演进，不允许长期停留在旧版角色模型

---

## 四、玩家信息可读性规范

### 9. 所有影响战斗决策的系统，必须有玩家可读出口

规则：

- 只要某个系统会影响玩家选择，就必须有明确的信息展示入口

包括但不限于：

- 被动技能
- 专精方向
- Buff / Debuff 效果
- 技能真实效果
- 成长选择内容
- 角色当前状态（如重伤、恢复中）

典型载体：

- 图鉴界面
- 升级界面
- Tooltip
- 状态栏
- 目标信息面板

禁止：

- 只有代码生效，但玩家无法理解其含义

---

## 五、技术债与原型清理规范

### 10. 旧原型方案有正式替代后，必须尽快降级或清退

规则：

- 原型期的临时实现可以存在，但不能长期与正式实现并存
- 一旦有稳定正式替代，就要尽快把旧链路降级为 fallback 或彻底移除

当前项目中的典型问题包括：

- 同时存在旧 `BattleUI` 逻辑和新 `BattleHudUI` 逻辑
- Prefab 驱动界面与 runtime builder 双轨并存
- 旧参数、旧路径、旧命名继续误导后续开发

目标：

- 避免双轨长期存在
- 降低维护和理解成本

---

## 六、系统设计收束规范

### 11. 后续开发以职业、专精、位置战术、成长构筑为核心

规则：

- 后续功能开发不能再以“补散件”为主
- 优先围绕以下主轴收束：
  - 职业身份
  - 专精分支
  - 位置战术
  - Buff / Debuff 联动
  - 成长构筑

要求：

- 数据结构服务成长
- UI 服务战斗决策
- 编辑器工具服务内容生产

禁止：

- 在核心未立住前大量扩泛内容
- 系统继续横向堆积但缺少中心主轴

---

## 七、推荐执行原则

后续开发中，默认按以下顺序判断实现方式：

1. 能否先落成可编辑的 Prefab / 资产？
2. 运行时是否只需要绑定和消费？
3. 是否会覆盖人工微调内容？
4. 玩家是否能理解这个系统的真实效果？
5. 这项改动是否服务职业 / 专精 / 构筑主轴？

如果答案不满足上述原则，应优先调整实现方案，而不是直接编码。

---

## 八、一句话总纲

**Prefab 决定结构和外观，编辑器决定生产方式，运行时只负责绑定与执行；所有核心战斗与成长系统都必须可配置、可读、可维护。**
